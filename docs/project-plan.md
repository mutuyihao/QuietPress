### AUD-051: 搜索 API SQL 注入

| 字段 | 值 |
|------|-----|
| **严重程度** | Critical |
| **状态** | Open |
| **领域** | 安全 |
| **影响范围** | `/api/search` 路由，所有公开搜索功能 |
| **责任人** | 后端工程师 |
| **计划修复** | 2026-06-02 |
| **代码验证** | ✅ 已确认 — `app/api/search/route.ts` 第20行 `.or(\`title.ilike.%${q}%,excerpt.ilike.%${q}%\`)` 直接插值 |

**问题描述：**
搜索 API 将用户输入 `q` 参数直接插入 Supabase `.or()` 过滤器，攻击者可构造恶意查询字符串操纵过滤器逻辑。

**修复步骤：**

1. 打开 `app/api/search/route.ts`
2. 替换直接字符串插值为安全方式：
   ```typescript
   const safeQuery = q.replace(/[%_]/g, '\\$&')
   query = query.or(`title.ilike.%${safeQuery}%,excerpt.ilike.%${safeQuery}%`)
   ```
3. 或创建 Supabase RPC 函数 `search_posts(query text)` 使用参数化 SQL
4. 同步修复 `lib/db/supabase/posts.ts` 中的搜索方法
5. 添加输入长度校验：`if (q.length > 200) return 400`

**验证方法：**
- [ ] 测试包含 `'`, `"`, `;`, `%`, `_` 的查询 — 不应导致错误或异常结果
- [ ] 测试 `'); DROP TABLE posts;--` — 应返回空结果而非错误
- [ ] 测试包含 `%` 通配符的查询 — 应被转义
- [ ] `pnpm lint` 和 `pnpm build` 无错误

### AUD-052: 评论内容 XSS 漏洞

| 字段 | 值 |
|------|-----|
| **严重程度** | Critical |
| **状态** | Open |
| **领域** | 安全 |
| **影响范围** | 所有博客文章评论区域 |
| **责任人** | 前端工程师 |
| **计划修复** | 2026-06-02 |
| **代码验证** | ✅ 已确认 — `app/api/comments/route.ts` POST 处理器未对 content 做 HTML 消毒；`comment-section.tsx` 渲染为文本但缺乏服务端防护 |

**问题描述：**
评论内容通过 POST 端点存储为原始文本，服务端未做 HTML 消毒。虽然当前渲染为纯文本，但未来引入富文本功能时可能导致 XSS。

**修复步骤：**

1. 在 `app/api/comments/route.ts` POST 处理器中添加 `sanitize-html` 消毒：
   ```typescript
   import sanitizeHtml from 'sanitize-html'
   const sanitizedContent = sanitizeHtml(content, {
     allowedTags: [],
     allowedAttributes: {},
   })
   ```
2. 在插入数据库时使用 `sanitizedContent` 替代 `content`
3. 在 `comment-section.tsx` 中确认评论内容以文本方式渲染（无 `dangerouslySetInnerHTML`）
4. 添加服务端内容长度校验（已有 5000 字符限制，验证是否强制执行）

**验证方法：**
- [ ] 提交含 `<script>alert('xss')</script>` 的评论 — 应渲染为纯文本
- [ ] 提交含 `<img src=x onerror=alert(1)>` 的评论 — 应被剥离或渲染为文本
- [ ] 提交含 `javascript:alert(1)` 链接的评论 — 应被中和
- [ ] 已有已审核评论仍正常显示
- [ ] `pnpm lint` 和 `pnpm build` 无错误
